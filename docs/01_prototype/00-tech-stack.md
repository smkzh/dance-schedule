# 00: 技術スタックの選定理由

---

## 使用技術一覧

| 技術 | 用途 |
|---|---|
| Next.js (App Router) | 画面 + APIルート |
| TypeScript | 型安全な開発 |
| Tailwind CSS | スタイリング・スマホ対応 |
| Supabase | データベース (PostgreSQL) |
| Vercel | デプロイ・URL公開 |

## Next.js

### 何をするもの？

React ベースの Web フレームワーク。「フレームワーク」とは、アプリを作るための土台・ルールのセット。

React 単体では「UI を作るライブラリ」に過ぎないが、Next.js を使うことで以下がまとめて揃う：

- **ファイルベースルーティング**: フォルダ構造がそのまま URL になる
- **サーバーコンポーネント**: データ取得をサーバー側で行い、HTML を生成してから送る
- **APIルート**: `route.ts` を置くだけでバックエンドの API エンドポイントを作れる

### ファイルベースルーティング

Next.js の App Router では、フォルダ名 = URL パスになる。

```
app/
├── page.tsx           → /
├── submit/
│   └── page.tsx       → /submit
├── numbers/
│   ├── page.tsx       → /numbers
│   └── [id]/
│       └── page.tsx   → /numbers/任意のID
└── api/
    └── submit/
        └── route.ts   → POST /api/submit
```

このルーティングの仕組みは **Next.js 固有の機能**。React や TypeScript は関係ない。

ファイル名にも意味がある：

| ファイル名 | 役割 |
|---|---|
| `page.tsx` | そのURLに対応するページ（ブラウザに表示） |
| `route.ts` | そのURLに対応するAPIエンドポイント（HTTPリクエストを処理） |

### 向いていること

- **サーバー + クライアントを1プロジェクトで書ける**: 別途バックエンドサーバーを立てる必要がない
- **データ取得をサーバー側で行える**: ページを開いたときにサーバーで Supabase に問い合わせ、結果を HTML にして送れる（サーバーコンポーネント）
- **デプロイが簡単**: Vercel との相性が最もよく、push するだけで公開できる
- **学習リソースが豊富**: ドキュメントやチュートリアルが充実している

### 向いていないこと

- **リアルタイム通信**: チャットや通知など、サーバーからデータを「プッシュ」したい場合は追加の仕組みが必要
- **単純な静的サイト**: ブログや LP など、データベースが不要な場合は Astro など軽量な選択肢がある
- **大規模なバックエンド処理**: 複雑なビジネスロジックや長時間の処理は、専用のバックエンドサーバーを別途立てた方が適している

### このプロジェクトで選んだ理由

- フロントエンド（画面）とバックエンド（APIルート）を1つのプロジェクトで管理できる
- Supabase へのデータ取得をサーバー側で行えるので、APIキーをブラウザに露出せずに済む
- Vercel へのデプロイが最も簡単

## TypeScript

### 何をするもの？

JavaScript に「型」を追加した言語。型とは「この変数は文字列」「この関数は数値を返す」のような情報。

```ts
// JavaScript（型なし）
function add(a, b) { return a + b; }

// TypeScript（型あり）
function add(a: number, b: number): number { return a + b; }
```

### 向いていること

- **バグの早期発見**: 型が合わないコードをエディタがエラーとして教えてくれる
- **補完の強化**: 変数のプロパティや関数の引数を自動で候補表示してくれる
- **コードの読みやすさ**: 型を見ると「何が入っているか」がわかる

### 向いていないこと

- **素早いプロトタイプ**: 型を書く手間があるため、試し書きには少し重い
- **型定義が複雑になりすぎる場合**: 無理に型をつけようとして `any` だらけになることがある

### このプロジェクトで選んだ理由

- Supabase から返ってくるデータの型を明示することで、プロパティ名のミスを防げる
- Web 開発を学ぶ上でも、型ありの方が後から他の言語への移行がしやすい

## Tailwind CSS

### 何をするもの？

CSSクラスを HTML に直接書くことでスタイルを当てるフレームワーク。

```tsx
// 通常のCSS（別ファイルに書く）
<div class="card">...</div>

// Tailwind（クラスに直接書く）
<div className="flex flex-col gap-4 p-4 border rounded-lg">...</div>
```

### 向いていること

- **CSSファイルを別途作らなくてよい**: HTML/JSX を見るだけでスタイルがわかる
- **スマホ対応が簡単**: `sm:` `md:` などのプレフィックスでブレークポイントを指定できる
- **デザインの一貫性**: あらかじめ定義された値（色・サイズ・余白）を使うのでバラつきにくい

### 向いていないこと

- **複雑なアニメーション**: CSS で細かく制御したい場合はクラスが煩雑になる
- **クラス名が長くなる**: 多くのスタイルを当てると className が長くなって読みにくい

### このプロジェクトで選んだ理由

- スマホ対応が `sm:flex-row` などのプレフィックスだけで書けて手軽
- CSSファイルを別管理する必要がなく、学習しながら書く量を減らせる

## Supabase

### 何をするもの？

PostgreSQL データベース・管理UI・API がセットになったサービス。

通常、データベースを使うにはサーバーを別途用意してSQLを直接実行するが、Supabase は JavaScript のクライアントライブラリを通じて直接テーブルを操作できる。

```ts
// テーブルからデータを取得する例
const { data } = await supabase.from("members").select("id, name");
```

### 向いていること

- **素早い立ち上げ**: テーブル作成・API・管理UIがすぐに揃う
- **ブラウザ上でデータ確認**: 管理ダッシュボードからテーブルの中身を直接確認・編集できる
- **無料枠が十分**: 今回の規模（数十人・数十テーブル）なら無料で動かせる

### 向いていないこと

- **大規模なトラフィック**: 無料枠には接続数・ストレージの上限がある
- **複雑なSQLが必要な処理**: Supabase のクライアントで書きにくいクエリは直接SQLを書く必要がある

### このプロジェクトで選んだ理由

- データベースの中身をブラウザで確認しながら開発できるため、デバッグが簡単
- 認証機能は今回使わないが、後から追加したくなったときに対応できる
- 無料プランで今回の規模は十分

## Vercel

### 何をするもの？

Next.js の開発元が提供するホスティングサービス。GitHub にプッシュするだけで自動デプロイされ、URL が発行される。

### 向いていること

- **Next.js との相性が最良**: Next.js の機能をすべて正しく動かせる
- **自動デプロイ**: `git push` するたびに本番環境が更新される
- **無料枠が十分**: 個人・小規模プロジェクトなら無料で運用できる

### 向いていないこと

- **常時起動のサーバーが必要な処理**: Vercel はリクエストごとに関数を起動する形式のため、WebSocket などの常時接続には向かない
- **大量のサーバーサイド処理**: 実行時間に制限がある

### このプロジェクトで選んだ理由

- メンバーは発行された URL をブラウザで開くだけで使える（アプリインストール不要）
- GitHub と連携して push するだけで本番に反映されるため、開発サイクルが速い
